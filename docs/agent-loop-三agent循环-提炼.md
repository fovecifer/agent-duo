# Claude Code 三 Agent 循环 与 Loop Engineering 提炼

> 一句话：演示用三个 agent（规划 / 构建 / 评判）循环迭代，约 40 分钟从零搭出一个完整可用的应用。核心观点——**真正的胜负不在于谁的模型更聪明，而在于谁的循环（loop）更好**。

- **链接**：https://x.com/AnatoliKopadze/status/2068690663919530207

  - Anthropic 工程师展示了从零搭建完整应用的过程，用了一个 **agent 循环**，约 **40 分钟**。
  - 三个角色分工：**plan（规划）→ build（构建）→ judge（评判）**，不断循环直到应用真正跑通。
  - 关键论断：*赢家未必用了最聪明的模型，而是搭出了最好的 loop。*
  - 痛点观察：AI 已普及多年，但大多数人每天仍在用**最慢的方式**——手动「提需求 → 等 → 改 → 再问」，全靠人肉驱动。

### 2. 配套工作坊：《Build Agents That Run for Hours》
- **链接**：https://www.youtube.com/watch?v=mR-WAvEPRwE
- 讲者：Ash Prabaker & Andrew Wilson，偏实操。
- 直接对应帖子里「judge」角色的核心：为什么**自评是陷阱**、对抗式独立评判更有效；为什么上下文压缩治不好「连贯性漂移」、而结构化交接可以。

#### 工作坊逐点提炼（含时间戳）

> 主题：如何构建能持续运行数小时（长周期）的 AI 智能体。

**① 长周期 agent 的三大核心挑战** `[02:36]`
- **上下文问题 (Context)**：容量有限（必须重置会话 → 「失忆」）；上下文腐坏（会话越深，逻辑连贯性越差）；以及**「上下文焦虑」**——模型逼近上限时会急躁地草草收尾。
- **规划能力不足 (Planning)**：不擅长长线规划，容易一次性想做完所有事，或建到一半就停。
- **无法准确评估自身输出**：存在**阿谀奉承（Sycophantic）**倾向，检查自己代码时往往盲目盖章通过，哪怕功能根本没实现。

**② 架构（Harness）的演进**
- `[08:01]` **Ralph Loop**：早期有效手段——不断把提示喂进 CLI、强制循环执行，直到满足退出条件。
- 随模型（Opus 4.6 / Sonnet 4.6）上下文管理与规划能力大幅提升，过去「每次任务都重置上下文」的复杂架构已被淘汰。
- `[34:21]` 现在支持高达 **100 万 token** 上下文 + **服务器端压缩（server-side compaction）**，单一会话可持续运行很久。

**③ 核心突破：生成器–评估器模式 (Generator–Evaluator)** `[18:37]`
借鉴 GAN（生成式对抗网络）思路，打造无需人类介入就能开发完整应用的 agent：
- **放弃自我检查，引入对抗压力**：训练一个「既能写又能自我批评」的模型很难，但单独调一个**严厉的独立评估器**很容易。
- **真实工具测试**：评估器不只是读代码，而是用 **Playwright** 等工具真的打开网页、点按钮、看报错，再把修改意见反馈给生成器。
- `[21:29]` **品味也能被评分**：很多人觉得设计/品味主观，但只要写下**带强观点的评分标准**（如：原创性、工艺、功能性、设计 四个维度），模型就能有效剔除「粗劣的 AI 感」审美。

**④ 规划师与契约协商 (Planner & Contracts)** `[23:47]`
完整应用开发需要三角色紧密配合：
- **规划师 (Planner)**：把用户一句话需求拆解成宏观的开发冲刺规范，但**不碰底层技术细节**——否则底层错误会在数小时后呈指数级放大。
- **契约协商 (Negotiation)**：写任何一行代码前，生成器与评估器必须通过读写 Markdown 文件**讨论并达成一致**，明确「什么样才算完成」（非常细粒度的验收标准）。评估器最终**只依据这份契约**验收。

**⑤ 给 AI 开发者的核心实践建议** `[38:55]`
- **自我评估是陷阱**：别让 agent 检查自己的工作，务必用对抗式的独立评估器。
- **用文件系统共享状态**：模型的总结往往是**有损的**，会导致任务偏离；推荐用文件系统作状态存储，建立结构化交接流程。
- **主观质量可以量化**：只要你对「理想状态」有清晰定义，就强制把它写成 **rubric（评分规则）**。
- **仔细阅读 Traces（运行日志）**：别只看结果——调试 agent 架构的唯一途径，就是像读报错栈一样**逐行读模型的心智过程和日志**，共情模型的痛点，据此优化提示词与架构。

> 注：演讲把最后一组建议表述为「五条」，本提炼捕捉到上面 4 条；若需要第 5 条可回视频 `[38:55]` 附近补全。

## 三、核心概念提炼（Loop Engineering）

### 范式转变：从「写 Prompt」到「写 Loop」
- 源头是 Boris Cherny 的一句话：**「我不再给 Claude 写 prompt 了，我的工作是写循环。」**
- Google 工程师 Addy Osmani 把这个模式命名为 **loop engineering（循环工程）**，在 X 上引发讨论。
- Boris 给的抽象层级递进：**source code（写代码）→ agent（让 agent 改代码）→ loop（用 /loop、/goal、routine 来驱动 Claude）**。
- 本质区别：一个 prompt 只给你**一次输出**；一个 loop 给你一个**可复利的、持续运转的操作**。

### plan-build-judge 三角色为什么有效
| 角色 | 职责 | 关键点 |
|------|------|--------|
| **Plan** | 拆解目标、制定方案 | 把抽象需求结构化 |
| **Build** | 实际写代码、执行 | 让它能像用户一样真正测试自己的产出，而不只是跑「测试」 |
| **Judge** | 独立评判是否达标 | **必须独立**——不能让构建 agent 自评，否则会放过自己的问题；对抗式评判更靠谱 |

### 验证（Verification）是循环的命门
- 循环只有在「Claude 能判断自己是否在取得进展」时才有用。
- 大量工程精力应花在**设计 agent 周围的环境**上：测试、反馈、verifier，让它能自我定位、自我纠错。
- 一个不带 verifier 的无人值守循环 = **一条非常自信地批量产出垃圾的传送带**。

### 记忆：把教训写进 CLAUDE.md
- Boris 第一原则：当 Claude **重复犯同样的错误**，就让它把教训写进 `CLAUDE.md` 或一个 skill。
- 逻辑：**纠正 → 写成持久指令 → 未来每次运行都更好**，而不是把修正困在某一次对话里。
- 注意：Claude 把 `CLAUDE.md` 当作**上下文（context）**，不是强制配置。
  - 「偏好某种测试模式」这类 → 写进 memory 即可；
  - 「禁止某个危险操作」这类 → 用 **PreToolUse hook** 强制拦截。

## 四、已实装的产品命令

这套理念现在已经做进 Claude Code 里了（一年前还需要自己写 bash 脚本维护）：

- **`/loop`**：本地重复执行某任务，最长可跑约 3 天。
- **`/goal`**：设定**完成条件**，Claude 持续工作直到条件为真。
  - 关键机制：**不让写代码的模型自己判断是否达标**，而是用一个**独立、更快的模型**在每一轮之后专门检查完成条件——这就是「judge」角色的产品化版本。
  - 例：`/goal all tests in test/auth pass and the lint step is clean`
  - 内部也叫 **Ralph loop**：每次试图停止时拦截，先自检条件，不满足就继续。
- **`/schedule`**：把任务丢到云端跑，合上笔记本也能继续，最长约一周。
- **`/workflows`**（动态工作流，research preview）：在 prompt 里提到「workflow」即可触发，Claude 自动生成编排计划，可在单会话中并行跑数百个 subagent。
- **Stop hook**：Claude 完成任务、准备交还控制权时触发的动作。可跑测试套件、打 CI 端点，失败就让模型继续修，不许停。

## 五、Addy Osmani 的三点风险提醒（重要）

Loop 是杠杆，但也可能放大问题。务必内建防护：

1. **Token 成本会剧烈波动**——agentic 循环里成本可能 10x 级别地复合增长，需要预算护栏（budget guard）。
2. **质量可能下降**——长循环里输出质量未必稳定。
3. **Slop（低质垃圾产出）风险真实存在**——没有 verifier 的无人循环尤其危险。

> 持久状态必须存在模型之外——文件、git、`CLAUDE.md`。模型本身是无状态的，再好的 prompt 也改变不了这一点。Loop engineering 是**系统工程**，不是「AI 技巧」。

## 五·补充、Loops 科普长文提炼（含广告甄别）

> 来源：Anatoli Kopadze 的 X 长文《Loops explained: Claude, GPT, Mira and what actually works》(2026-06-20)。
> **甄别提示**：本文前半是扎实的 loop 科普，组织清晰；**后半段是对 Telegram 产品「Mira」的软广**，并带关注/订阅 CTA。下面只收录前半段有价值的框架，Mira 部分仅作记录、不作技术建议。

### 五阶段循环解剖
`DISCOVER`（弄清要做什么）→ `PLAN`（决定怎么做）→ `EXECUTE`（动手）→ `VERIFY`（对照目标校验）→ `ITERATE`（没达标就把结果喂回去重来）。
三个关键：
- **Verify 是心脏**：没有对结果的真实校验，就不是循环，而是「agent 反复同意自己」。校验可以是硬测试、可度量条件，或一份评分 rubric。**没有闸门 = 让做题的人自己改卷**，而它打分太宽松。
- **State 让循环会学习**：每一轮要记住已试过什么，否则会永远重复同一个错。在旁边维护一份小记录（已完成 / 失败了 / 下一步），明天接着跑而非从零开始——这也正是成本开始飙升的地方。
- **Stop condition 让它别失控**：每个正经循环都要有两条退出路径——成功，以及硬上限（如「8 次后停下并报告」）。没有它，就是一台能空转一整夜的烧钱机器。

### 「要不要做（重型）循环」的四盒判据
四个**全部满足**才值得上重型循环；缺一个就退回一个好 prompt：
1. 任务**至少每周重复**（一次性的不值回设置成本）；
2. 有东西能**自动否决坏输出**（测试 / 类型检查 / 构建 / linter / 硬规则）；
3. agent 能**端到端独立完成**，而不是把一半甩回给你；
4. 「完成」是**客观**的，不是品味判断（靠主观质量时人类仍然更强）。

> 诚实结论：loop engineering 是真的，但大多数人**还不需要重型版**；人人能用的是轻量版（见下方手搓循环）。

### 循环的五个构件（Claude Code / Codex 现已全部内置）
1. **自动触发（心跳）**：定义 prompt + 节奏 + 目标，按计划自动跑。对应 `/loop`、`/goal`、hooks，或推到 cron / GitHub Actions。
2. **Skill（可复用指令）**：把规则、模式、「绝对不能碰的清单」存成文件，循环每次读取，避免指令烂在某个没人维护的计划里。
3. **子 agent（让制造者远离检查者）**：把「做活的 agent」和「检查的 agent」分开——写代码的那个给自己改卷太宽松；第二个 agent（不同指令、有时用更强模型 + 更高 effort）能抓出第一个自我说服放过的问题。**写手可以快而便宜，审查者要慢而严格——质量大半来自这层分离。**
4. **Connectors（让它行动而非建议）**：区别在于「给你一句修复建议」vs「自己开 PR、关联 ticket、构建变绿后在频道里 @ 你」。
5. **Verifier（闸门）**：自动否决坏活的测试 / 类型检查 / 构建。**这一块决定循环是帮你还是只烧钱，其余都是管道。**

### 成本：没人提的那一面
- 每一轮 agent 都要**重读全部上下文**（目标 + 代码 + 上轮结果 + 失败记录），这堆东西每轮都在变大并重新过一遍模型。跑 10 次 ≠ 10 个 prompt，而是 **10 个越来越大的 prompt**。制造者–检查者那招提升质量的同时也**让账单翻倍**。
- **真正该追踪的指标**：不是花了多少 token，而是**「每个被采纳改动的成本」(cost per accepted change)**。如果给你 10 个结果你扔掉 6 个，你就在做它本该替你省下的审查工作——**采纳率低于 50%，就得不偿失**。
- **Ralph Wiggum loop**（Geoffrey Huntley 命名）：agent 过早判定完成、半成品就退出，循环继续空转烧钱。**没有能让活计「失败」的硬闸门，循环不会崩——它会无声地给你计费。**

### 正确的搭建顺序（顺序比工具更重要）
1. 先让**一次手动运行**稳定可靠；
2. 把它**存成 skill**（固化指令）；
3. 用 loop **包起来**（加闸门 + 停止条件）；
4. **最后**才上定时。
> 跳步去定时一个你还没手动验证过的东西，就是循环在你睡觉时炸掉的标准死法。先证明一次，再加固，再自动化。

### 轻量版：在任意 LLM 里手搓一个自检循环
不需要 coding agent，一段 prompt 就能在 Claude / ChatGPT 里跑一个手动循环。要点是**一次性给模型三样东西**：目标、严格的成功标准、一个强制它在「允许停止」前先自检的协议。（下面是改写版模板）

```
你将以循环方式工作，直到产出达到标准。

任务：
[精确描述你要的产出]

成功标准（从严，不许放水）：
- [标准 1]
- [标准 2]
- [标准 3]

循环协议（每一轮都重复）：
1. PLAN   —— 说明下一步唯一动作
2. DO     —— 产出或改进工作
3. VERIFY —— 对每条标准打 1–10 分，残酷诚实，列出仍然薄弱之处
4. DECIDE —— 每条都 ≥8 则打印「FINAL」并停止；
            否则打印「ITERATING」，优先修最弱的一项再来一轮

规则：
- 任一标准 <8 绝不算完成
- 每轮必须修上一次 VERIFY 里分数最低的那项
- 不要反问我；做出合理假设、注明、继续

开始。一直循环到 FINAL。
```

> 但注意它缺什么——**你就是触发器**：你开了对话、贴了 prompt、坐在那看它迭代；关掉标签页它就没了，没有定时、没有事件触发、它够不到你。要让循环自己跑、按时跑、被真实事件触发，才需要前面那套重型世界（工具 / 托管 / 闸门 / 账单）。

### 关于后半段的 Mira（仅记录，非背书）
文章后半把上述概念导向一个 Telegram 产品 **Mira**：宣称通过 Composio 连 500+ 应用、跨会话长期记忆、模型无关（GPT/Claude/Gemini），用自然语言描述即生成「Skill」（即带触发器+动作+自运行的循环）。
- **判断**：这是软广，claim 未经独立核实。其中「监控机票价格、到价自动**购买**」这类**自主下单/花钱**的能力尤其要谨慎——这类操作风险高，宜人工确认。
- **对你的实际意义**：作者自己的结论也是「先用现成免费的，真不够用了再考虑上重型」。对你而言，重型那套你本就在 Claude Code / Codex 里自建，Mira 不是你的方向；但它「**用自然语言描述 = 触发器+动作+自运行**」的产品化封装思路，可作为你设计 `agent-duo` 编排接口时的参考。

## 六、落地到我自己的环境（Claude Code + Codex / agent-duo）

结合之前做的 `agent-duo`（tmux + iTerm2 标签页里的 CLI agent 点对点通信），这套 plan-build-judge 循环可以这样映射：

- **Plan / Build**：可由 Claude Code 主会话或 Codex 承担；
- **Judge**：用 `/goal` 设定明确的完成条件，让**独立模型**做评判，而不是让构建方自评；
- **持久记忆**：教训写进各 repo 的 `CLAUDE.md`；
- **防护**：加 budget guard + Stop hook 跑真实测试，避免无人循环跑飞或烧 token。

**待办 / 可深入**：
- [ ] 在现有 `agent-duo` 上加一个独立 judge agent（对抗式评判）
- [ ] 用 `/goal` + Stop hook 配一套「测试通过才退出」的循环
- [ ] 给循环加 token 预算护栏，防止成本复合膨胀
